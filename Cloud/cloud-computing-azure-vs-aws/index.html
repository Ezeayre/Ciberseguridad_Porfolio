Guía interactiva: AWS vs Azure - Servicios, Seguridad y Buenas Prácticas

📚 Introducción

Cloud Computing ha revolucionado la forma en que las empresas y desarrolladores gestionan sus infraestructuras. En lugar de comprar y mantener servidores físicos, ahora puedes alquilar recursos computacionales bajo demanda, pagando solo por lo que usas.

Los dos líderes del mercado son AWS (Amazon Web Services) y Azure (Microsoft). Ambos ofrecen cientos de servicios, pero tienen diferencias importantes en enfoque, ecosistema y curva de aprendizaje.

Esta guía te ayudará a entender los conceptos fundamentales, los servicios principales de cada plataforma, y las mejores prácticas de seguridad para implementar en la nube.

🛡️ Seguridad en Cloud

🤝 Modelo de Responsabilidad Compartida

En cloud, la seguridad es responsabilidad compartida entre el proveedor (AWS/Azure) y tú. El proveedor se encarga de la seguridad DEL cloud, tú de la seguridad EN el cloud.

¿Qué significa esto?

  • Proveedor (AWS/Azure): Seguridad de los datacenters, hardware, red física, encriptación de datos en reposo por defecto
  • Tú: Seguridad de tus datos, configuración de firewalls, gestión de usuarios, parcheo de sistemas, encriptación de datos sensibles

💡 Ejemplos del Día a Día

IaaS - Como alquilar una computadora vacía:

Es como si alquilas una PC pero viene sin Windows ni programas. Tú instalas el sistema operativo, los programas, configurás todo. El proveedor solo te da el hardware. Ejemplo técnico: EC2, Azure VMs

PaaS - Como Netflix o Spotify:

Netflix maneja toda la infraestructura (servidores, bases de datos, streaming). Tú solo subes tu película y configuras quién puede verla. No te preocupás por servidores ni parches. Ejemplo técnico: Lambda, Azure Functions, RDS

SaaS - Como Gmail, Google Docs, Office 365:

Google maneja TODO: servidores, seguridad, actualizaciones, backups. Tú solo creás tu cuenta, escribís emails y gestionás quién tiene acceso. Pero aún sos responsable de no compartir tu contraseña ni subir datos confidenciales.

🎮 Otros ejemplos que usás todos los días:
  • Instagram: SaaS - Ellos manejan todo, vos solo subís fotos y gestionás tu cuenta
  • WhatsApp Web: SaaS - Meta maneja servidores y seguridad, vos solo chateás
  • Dropbox / Google Drive: SaaS - Ellos guardan tus archivos, vos los subís y compartís
  • Spotify: PaaS/SaaS - Ellos manejan streaming, vos solo escuchás y creás playlists

⚠️ Regla de oro: Nunca asumas que el proveedor hace todo. Siempre verifica qué es tu responsabilidad.

💻 Comandos CLI Básicos - AWS

Comandos esenciales de AWS CLI para tareas diarias. Requiere aws configure previo con tus credenciales.

🔶 EC2 - Máquinas Virtuales

aws ec2 describe-instances
Lista todas las instancias EC2 en tu cuenta. Muestra estado, IPs, tipo de instancia y más.
aws ec2 start-instances --instance-ids i-1234567890abcdef0
Inicia una instancia EC2 específica. Reemplaza el ID con tu instancia real.
aws ec2 stop-instances --instance-ids i-1234567890abcdef0
Detiene una instancia EC2 específica. Los datos en disco persisten, pero dejas de pagar por compute.
aws ec2 run-instances --image-id ami-0abcdef1234567890 --count 1 --instance-type t2.micro
Crea una nueva instancia EC2. Necesitas especificar el AMI (imagen), cantidad y tipo de instancia.

📦 S3 - Almacenamiento

aws s3 ls
Lista todos los buckets S3 en tu cuenta.
aws s3 mb s3://mi-bucket-unico
Crea un nuevo bucket S3. El nombre debe ser globalmente único.
aws s3 cp archivo.txt s3://mi-bucket/archivo.txt
Copia un archivo local a un bucket S3.
aws s3 sync ./carpeta-local s3://mi-bucket/carpeta
Sincroniza una carpeta local con S3. Solo sube archivos nuevos o modificados.

🗄️ RDS - Base de Datos

aws rds describe-db-instances
Lista todas las instancias de base de datos RDS.
aws rds start-db-instance --db-instance-identifier mi-db
Inicia una instancia de base de datos RDS que estaba detenida.
aws rds stop-db-instance --db-instance-identifier mi-db
Detiene una instancia de base de datos RDS. Ahorra costos cuando no la usas.

🔐 IAM - Control de Acceso

aws iam list-users
Lista todos los usuarios IAM en tu cuenta.
aws iam create-user --user-name nuevo-usuario
Crea un nuevo usuario IAM. Luego necesitas asignarle permisos.
aws iam attach-user-policy --user-name nuevo-usuario --policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess
Asigna una política a un usuario. Este ejemplo da permisos de solo lectura.

💻 Comandos CLI Básicos - Azure

Comandos esenciales de Azure CLI para tareas diarias. Requiere az login previo.

🔵 Virtual Machines

az vm list
Lista todas las máquinas virtuales en tu suscripción de Azure.
az vm start --name mi-vm --resource-group mi-grupo
Inicia una máquina virtual específica. Necesitas el nombre del VM y del resource group.
az vm stop --name mi-vm --resource-group mi-grupo
Detiene una máquina virtual. Dejas de pagar por compute pero sigues pagando por el disco.
az vm create --resource-group mi-grupo --name mi-vm --image UbuntuLTS --admin-username azureuser
Crea una nueva máquina virtual. Especifica resource group, nombre, imagen y usuario admin.

📦 Storage

az storage account list
Lista todas las cuentas de almacenamiento en tu suscripción.
az storage account create --name mistorage123 --resource-group mi-grupo --location eastus
Crea una nueva cuenta de almacenamiento. El nombre debe ser único globalmente (3-24 caracteres, minúsculas).
az storage blob upload --container-name mi-contenedor --name archivo.txt --file archivo.txt --account-name mistorage123
Sube un archivo a un contenedor de blob storage.

🗄️ SQL Database

az sql db list --resource-group mi-grupo --server mi-servidor
Lista todas las bases de datos SQL en un servidor específico.
az sql db create --resource-group mi-grupo --server mi-servidor --name mi-db --edition GeneralPurpose
Crea una nueva base de datos SQL. Especifica resource group, servidor, nombre y edición.

🔐 Entra ID (Azure AD)

az ad user list
Lista todos los usuarios en Azure Active Directory (Entra ID).
az role assignment list --assignee usuario@dominio.com
Lista todos los roles asignados a un usuario específico.

💡 Tips Importantes y Buenas Prácticas

✅ SIEMPRE HACER:
  • Usar Multi-AZ para alta disponibilidad
  • Encriptar datos sensibles
  • Usar IAM roles (no Access Keys)
  • Monitorear costos (puede sorprenderte)
  • Hacer backups regulares
  • Usar Security Groups restrictivos
  • Habilitar MFA en tu cuenta
  • Usar tags para organizar recursos
  • Documentar arquitecturas y cambios
  • Probar en entorno de desarrollo primero
❌ NUNCA HACER:
  • Dejar puertos abiertos innecesariamente
  • Usar credenciales root en producción
  • Guardar secretos en código
  • No revisar facturas (cloud es caro)
  • Olvidar eliminar recursos (gastas igual)
  • Confiar en una sola AZ/región
  • Guardar contraseñas en texto plano
  • Compartir credenciales por email/chat
  • Ignorar alertas de seguridad
  • Usar servicios sin entender el costo
